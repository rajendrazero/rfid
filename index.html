Tujuan & Requirements

Tujuan Aplikasi

Membangun sistem manajemen troli bandara berbasis RFID untuk meningkatkan efisiensi pelacakan dan manajemen troli di lingkungan bandara.

  • Mengurangi waktu pencarian troli oleh petugas
  • Memantau ketersediaan troli di berbagai lokasi bandara
  • Memberikan data real-time tentang status troli
  • Meningkatkan pengalaman pengguna dengan troli yang mudah ditemukan
  • Memudahkan manajemen inventaris troli

Functional Requirements

  1. Sistem login dengan role user dan admin
  2. Pelacakan lokasi troli melalui RFID
  3. Fitur CRUD (Create, Read, Update, Delete) untuk troli
  4. Pemberian status troli (tersedia, rusak, dalam perbaikan)
  5. Monitoring user aktif dan aktivitas mereka
  6. Dashboard informasi troli dengan detail lengkap
  7. Laporan historis pergerakan troli

Contoh Penggunaan

Mencari troli dengan lokasi tertentu, status tertentu, kode TR-01, menampilkan 10 item per halaman di halaman ke-2

Informasi Visualisasi 3D

Koordinat lokasi 3D (x_coordinate, y_coordinate, dan z_coordinate) dapat digunakan untuk visualisasi peta 3D lokasi troli di aplikasi:

  • Menampilkan posisi troli pada peta 3D bandara
  • Visualisasi distribusi troli per lantai/level
  • Heatmap kepadatan troli di area bertingkat
  • Rute pergerakan troli antar lantai
  • Identifikasi area dengan kebutuhan troli mendesak berdasarkan lantai

Potensi Kesalahan

  • Menggunakan format UUID tidak valid untuk location_id atau status_id
  • Menggunakan nilai di luar range 20 untuk limit
  • Menggunakan nilai negatif untuk page
  • Filter yang terlalu spesifik sehingga tidak ada troli yang cocok
  • Lokasi tanpa koordinat Z akan ditampilkan sebagai 0.0 (lantai dasar)

Penggunaan Z-Coordinate

Nilai z_coordinate merepresentasikan level/lantai troli:

  • 0.0: Lantai dasar/Ground floor
  • 1.5: Lantai 2 (ketinggian 1.5 meter dari dasar)
  • 3.0: Lantai 3
  • -1.0: Basement/Lantai bawah tanah

Sistem dapat memfilter troli per lantai dengan query tambahan: GET /trolleys?z_coordinate=1.5

Pelacakan 3D

Dengan penambahan koordinat Z, sistem sekarang mendukung pelacakan 3D:

  • z_coordinate: Menentukan level/lantai (0 = lantai dasar, 1.5 = lantai 2, dst)
  • z_offset: Akurasi posisi vertikal relatif terhadap titik referensi
  • Memungkinkan visualisasi 3D pergerakan troli di area bertingkat
  • Mendeteksi perpindahan troli antar lantai melalui lift/eskalator

Best Practices 3D

  • Gunakan z_coordinate = 0 untuk lantai dasar, 1.5 untuk lantai 2, 3.0 untuk lantai 3
  • Kalibrasi sensor RFID secara berkala untuk akurasi 3D
  • Gunakan z_offset untuk menentukan posisi relatif di tangga/lift
  • Integrasikan dengan model digital twin 3D bandara

                "new_role": "Admin"
            }
            }

Response (Error 400)

{
            "status": "error",
            "message": "role_id harus berupa UUID yang valid",
            "error": 400
            }

Terjadi ketika role_id tidak valid atau tidak sesuai format UUID

Response (Error 401)

{
            "status": "error",
            "message": "Hanya admin yang dapat mengubah role user",
            "error": 401
            }

Terjadi ketika user yang mencoba mengakses bukan admin

Response (Error 404)

{
            "status": "error",
            "message": "User dengan ID tersebut tidak ditemukan",
            "error": 404
            }

Terjadi ketika user dengan ID yang diberikan tidak ditemukan

Response (Error 500)

{
            "status": "error",
            "message": "Gagal memperbarui role user",
            "error": 500
            }

Terjadi ketika ada masalah server internal

Contoh Penggunaan

PATCH /users/123e4567-e89b-12d3-a456-426614174000/role

Mengubah role user dengan ID tertentu

Contoh Request:

{
                "id": "gwefuegf-2144-a7dfb-schsdvhdvhdve",
                "role_id": "f1aabf12-9211-4e2b-a7b9-8126cf2a7df0"
            }

Keamanan dan Izin

  • Hanya user dengan role admin yang dapat mengakses endpoint ini
  • Setiap permintaan harus menyertakan token akses yang valid
  • Log aktivitas perubahan role harus dicatat untuk audit
  • Verifikasi ganda disarankan untuk perubahan role penting

Penting

  • Pastikan role_id yang dikirim valid dan sudah terdaftar di sistem
  • User tidak dapat mengubah role sendiri menjadi non-admin
  • Perubahan role dapat mempengaruhi akses pengguna secara signifikan
  • Lakukan verifikasi tambahan untuk perubahan role ke admin

Tahapan Pembuatan API

1

Setup Autentikasi

Fundamental

Implementasi sistem keamanan dan manajemen pengguna dengan token management berbasis database.

Endpoint Prioritas

  • POST /auth/register - Pendaftaran user/admin
  • POST /auth/login - Login dengan JWT + token DB
  • POST /auth/refresh - Refresh token + validasi DB
  • POST /auth/logout - Revoke session + revoke DB
  • POST /auth/logout-all - Terminate all sessions Baru

Workflow Implementasi

  1. Setup database user dan user_tokens
  2. Implementasi JWT authentication
  3. Role-based access control
  4. Token management dengan penyimpanan refresh token di DB
  5. Implementasi revoke token dan logout semua perangkat

Best Practice

Token Management: Selalu simpan refresh token di database untuk:

  • Validasi server-side pada proses refresh token
  • Kemampuan revoke token per perangkat
  • Logout dari semua perangkat sekaligus
  • Proteksi terhadap token yang dicuri

Gunakan middleware otentikasi untuk melindungi endpoint sensitif dan verifikasi status token di database.

Pastikan is_logged_in: Selalu di ubah sesuai kondisi login atau tidak

2

Manajemen Troli

Core Business Logic

Membangun sistem CRUD troli dasar sebagai inti aplikasi manajemen troli.

Endpoint Prioritas

  • POST /trolleys - Buat troli baru
  • GET /trolleys - Ambil daftar troli (filter/pagination)
  • GET /trolleys/{id} - Detail troli spesifik
  • PUT /trolleys/{id} - Update data troli
  • DELETE /trolleys/{id} - Hapus troli

Struktur Data 

Troli {
            id: UUID
            rfid_code: String
            trolley_code: String
            status_id: UUID (FK)
            created_at: DateTime
            updated_at: DateTime
        }

Penting

Implementasikan validasi RFID unik untuk mencegah duplikasi. Gunakan soft delete untuk riwayat historis.

3

Monitoring Perangkat RFID

Infrastructure

Sistem ini mencatat dan memantau status perangkat pembaca RFID secara real-time termasuk status aktif dan lokasi terakhir terhubung.

Endpoint Prioritas

  • GET /readers - Daftar semua RFID reader
  • PATCH /readers/:id - Update status atau lokasi reader
  • GET /readers/active - Daftar reader yang aktif (opsional)
  • DELETE /readers/:id - Nonaktifkan perangkat reader (opsional)

Workflow Implementasi

  1. Setup tabel rfid_readers di database
  2. Hubungkan setiap RFID reader ke lokasi fisik (via location_id)
  3. Buat API untuk memonitor status dan IP address
  4. Integrasi ke dashboard monitoring
  5. Implementasi heartbeat checker untuk status last_online

Best Practice

Standard Identifikasi Perangkat: Gunakan device_code yang konsisten seperti Reader-Terminal3A atau Reader-GateD1 untuk memudahkan tracing.

  • Pantau status aktif/inaktif berdasarkan data last_online dan heartbeat interval
  • Simpan IP address untuk keperluan audit jaringan dan troubleshooting
  • Integrasikan notifikasi jika device tidak aktif dalam periode tertentu

Gunakan middleware admin-only untuk mengakses endpoint ini agar tidak disalahgunakan oleh user biasa.

4

Pelacakan Lokasi

Core RFID Functionality

Implementasi sistem pelacakan berbasis RFID untuk memonitor pergerakan troli secara real-time.

Endpoint Prioritas

  • POST /trolleys/{id}/location-logs - Catat lokasi baru
  • GET /trolleys/{id}/logs - Riwayat lokasi troli
  • GET /trolleys/location/{rfid} - Lokasi terkini via RFID

Workflow RFID

  1. Troli terdeteksi oleh RFID reader
  2. Kirim lokasi ke endpoint API
  3. Update database dengan lokasi baru
  4. Tampilkan di peta real-time

Integrasi Hardware

Pastikan RFID reader dikonfigurasi untuk POST request ke endpoint lokasi-logs setiap kali troli terdeteksi. Gunakan koordinat offset untuk akurasi posisi.
